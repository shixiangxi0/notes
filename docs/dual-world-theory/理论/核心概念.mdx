# 核心概念

双世界理论为复杂系统提供了一种不同于传统面向对象的建模方式。它的核心主张是：把系统理解为一个在时间中持续演化的过程，而不是一组在空间中被组织起来的对象。

## 事实即关系变化

传统建模默认事实是状态：

```text
Player.hp = 100
Player.x = 123
```

但状态本身往往只是计算材料。真正的事实是关系变化：

```text
Player 进入 BossRoom
Player 命中 Enemy
Player 获得 Sword
Player 死亡
```

坐标 `x=123` 不是事实，「越过终点线」才是。血量数值不是事实，「受到伤害」才是。

双世界理论采用关系本体论，而非状态本体论。因果层维护的不是对象属性，而是被裁决并进入历史的关系变化。

## 面向因果

**面向因果**指的是在建模时，以事件、事实、因果与裁决过程作为系统组织的最小基础。

这与面向对象形成对照：

| 面向对象 | 面向因果 |
|---------|---------|
| 以对象、类、方法为最小单元 | 以事件、事实、裁决为最小单元 |
| 自底向上：先划分对象，再处理时间 | 自顶向下：先定义事实如何形成，再派生表现 |
| 状态分散在对象私有空间中 | 关系事实集中在因果层，感知层持有投影与局部状态 |

面向因果尤其适用于那些需要长期运行、持续交互、不断吸收变化的系统，例如操作系统、3A 游戏、交易系统、在线协作工具。

## 两层模型

双世界理论把系统拆分为两个相互协作的层次：

工程上看，**因果层是一个关系模型**：它维护的是经过裁决的关系变化（事件历史 + 裁决规则）。**感知层是一个局部状态模型**：它维护的是事实的派生状态、本地预测、连续表现和临时反馈。

### 因果层

因果层是系统的**事件解释与裁决引擎**。它不是状态数据库，而是把外部事件转化为被承认事实的解释机制。

因果层维护的核心资产是：

- **事件历史**：进入系统的语义事件序列
- **裁决规则**：解释事件、解决冲突、计算结果的规则集合

事实由事件和规则派生：

$$
Fact = Rule(Event\ History, Event)
$$

一旦产出，事实又成为下一轮裁决的输入，并进入可解释的历史。

因果层面对的是 `order_submitted`、`payment_confirmed`、`attack_requested` 这样的业务语义，而不是按钮、鼠标、页面控件这些外观。

因果层的核心机制是 $F_t$，也称为**解释函数**或**裁决机制**。它不是一个单一函数，而是一个**规则集合 + 解释引擎**。它回答的问题是：新事件如何被纳入已有世界，并转化为新的事实结构。

$$
World(t+1) = F_t(World(t), Event)
$$

其中：

- $World$ 是系统在时刻 $t$ 的世界状态，可进一步分解为 **事实（Facts）** 与 **因果结构（Causality）** 两部分
- $Event$ 是进入系统的新变化
- $F_t$ 是时刻 $t$ 的解释规则集合

### 裁决规则网络

$F_t$ 不是一条规则，而是一个规则网络。

一个事件可能触发多条规则；一条规则也可能依赖多个事件才能触发。规则产出的事实又可能成为新规则的输入。

```text
Event A ─┬→ Rule 1 ─→ Fact X
         ├→ Rule 2 ─→ Fact Y
         └→ Rule 3 ─→ Fact Z

Rule 1 ←┬─ Event A
        └─ Event B
```

这个多对多关系构成因果层的核心处理结构。

### 感知层

感知层是世界事实的响应式投影。它负责：

- 消费事实并形成投影
- 检测外部变化
- 把外部变化整理成语义事件送回因果层
- 组织连续表现、局部反馈和用户体验

感知层可以有自己的局部状态，例如页面展开、动画进度、通知是否已读、摄像机补间。这些状态是事实的派生，本身不构成世界事实。

这种局部状态与现代前端组件中的 local state 类似：它让组件或感知节点能够自治地响应变化、组织连续反馈，而不必事事回源到权威状态。但 local state 不能替代全局事实；一旦它可能影响未来世界事实，就必须通过语义事件提交给因果层裁决。

因果层为了裁决，也会临时折叠出派生状态（如当前 HP、当前 Buff 列表）。这些状态同样不是事实，而是用于下一次裁决的辅助数据。

### 两层的关系

```text
外部输入 → 感知层检测 → 翻译成语义事件 → 因果层裁决 → 产生新事实 → 感知层消费事实 → 呈现给外部
```

感知层不直接改写世界事实，只能通过语义事件请求因果层裁决；因果层不直接处理表现，只发布事实供感知层订阅。

## 关系事实与局部状态

工程上可以把两层理解为两个不同的数据模型：

| | 因果层（关系事实） | 感知层（局部状态） |
|---|---|---|
| 例子 | `damage_applied`、`player_entered_room`、`trade_executed` | 动画进度、摄像机位置、hover 状态、预测位移 |
| 是否进入历史 | 是 | 否 |
| 是否影响未来裁决 | 是 | 否 |
| 是否可被覆盖/丢弃 | 否（进入历史，只能被新事实演进，不能由感知层随意丢弃） | 是（新事实到达后必须让位） |
| 是否需要可追溯 | 是 | 否 |

简单地说：因果层关心“发生了什么关系变化”，感知层关心“这个变化在我这边看起来、感觉起来如何”。

## 因果层的三个基本属性

因果层必须满足三个基本属性：

1. **可裁决性（Adjudicability）**：当多个事件冲突、并发或矛盾时，规则必须能给出唯一事实版本。
2. **确定性（Determinism）**：给定相同的初始事实和相同的事件序列，因果层总是产生相同的新事实序列。
3. **可追溯性（Traceability）**：每个事实都能回追到触发它的事件和规则。

最终因果一致性是这三个属性的综合结果。

## 判断该放哪一层

最实用的判断方式只有一个问题：

> **这件事一旦发生，会不会影响未来的世界事实？**

如果会，它就属于因果层；如果不会，它就留在感知层。

更进一步：

> **这件事是否需要进入历史、被追溯、被多个节点共享？**

如果需要，它属于因果层；如果只是某个感知节点的本地反馈，它属于感知层。


## Headless 与 AI 友好

整个项目应该围绕因果层构建一个最小逻辑视图。通过 Headless 模式运行项目，可以天然地对 AI 和自动化测试友好：

- 因果层可以在没有渲染、没有 UI 的情况下独立运行
- 测试可以直接向因果层输入事件，验证事实输出
- AI 代理可以通过语义事件与世界交互，而不需要模拟鼠标键盘

## 与其他概念的关系

- **因果层不等同于 Event Sourcing**：Event Sourcing 的前提是事件已经是事实，系统只负责存储和回放。但因果层把事件当作需要解释的输入：事件可能冲突、可能无效、需要被规则裁决后才能成为事实。因果层可以使用 Event Sourcing 作为存储技术，但 Event Sourcing 本身不包含解释、裁决和一致性保证。
- **感知层不等同于 MVC 的 View**：View 通常是被动的，而感知层还负责检测输入、切割事件、组织连续反馈，在不影响世界事实的情况下可以连续自治。
- **最终因果一致性不是瞬时强一致**：它允许滞后和局部不一致，但要求最终收敛到单一可叙述的事实版本，类似于分布式中的最终一致性。
