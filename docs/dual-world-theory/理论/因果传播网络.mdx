# 因果传播网络

> **事件需要裁决，事实需要传播。**

因果层接收外部事件，将事件裁决为事实，再让事实传播产生新事件，最终形成可追溯的历史版本。

---

## 事件

事件是外部系统提交的变更请求。它表达了「希望某事发生」，但尚未经过裁决。

游戏中：

```text
attack_requested
move_requested
spell_cast_requested
```

数据库中：

```text
UPDATE account SET balance = balance - 100
```

操作系统中：

```text
进程请求 CPU 时间片
设备发出中断
```

事件本身不是事实。例如两个玩家同时拾取一个物品：

```text
pick(A, item)
pick(B, item)
```

两个事件都进入系统，但不能同时成为事实。

---

## 事实

事实是经过裁决后，进入历史的关系变化。

```text
damage_applied
hp_depleted
entity_dead
cpu_assigned
trade_executed
```

事实不是状态快照。`Player.hp=100` 是状态，「Player 受到致命伤害」才是事实。坐标 `x=123` 是状态，「Player 越过终点线」才是事实。

事实具有三个特点：

- **不可否认**：一旦进入历史，它就是当前世界的权威依据。
- **可追溯**：每一个事实都能回答「为什么发生」「由哪个事件触发」「经过哪些规则裁决」。
- **可派生**：事实本身可以通过传播规则产生新的事件。

---

## 裁决与传播

因果层通过两类规则运行：

- **裁决规则**：决定事件能否成为事实。
- **传播规则**：决定事实会产生哪些新事件。

核心循环：

```text
事件 → 裁决 → 事实 → 传播 → 新事件 → ...
```

以游戏里的一次攻击为例：

```text
玩家按下攻击键
        ↓
attack_requested（事件）
        ↓
裁决：目标在范围内？技能冷却？玩家未眩晕？
        ↓
damage_applied（事实）
        ↓
传播：敌人 HP 是否 <= 0？
        ↓
裁决：entity_died?（新事件）
        ↓
entity_dead（事实）
        ↓
传播：生成掉落物、更新任务、清除仇恨
```

## 复杂性的根源：因果传播网络

如果没有时间，世界只是一个静态关系结构；一旦引入时间，世界就变成不断运行的因果传播网络：

```text
静态结构（无时间）                因果传播网络（有时间）

Object                            Event
  |                                 ↓
 State                            Fact
  |                                 ↓
 Relation                          Event
                                    ↓
                                   Fact
                                    ↓
                                   ...
```

左侧的复杂度只来自规模：对象多少、状态多少、关系多少。这是**空间复杂度**。

右侧的复杂度不再来自“有什么”，而来自：

> **变化如何传播。**

对象、状态、模块只是承载这些传播关系的空间结构。软件复杂，本质上是因为我们必须在持续变化的因果网络中管理四个问题：

- 一个变化会影响多少地方？
- 一个结果依赖哪些条件？
- 传播什么时候停止？
- 新的变化如何中断旧传播？

这也是为什么传统面向对象在这类系统中会显得吃力：OOP 擅长回答“谁拥有数据”、“谁属于谁”——这些都是静态空间关系；但复杂软件真正需要回答的是动态因果关系。

### 复杂度的四个来源

因果传播网络中，几乎所有的工程困难都可以归为四种基本拓扑问题。

#### 1. 分叉（Branching）

一个事实可能影响多个未来事件：

```text
Fact
  |
  +--> Event A
  |
  +--> Event B
  |
  +--> Event C
```

它带来的问题是：**一个变化会影响多少地方？**

游戏中：

```text
entity_dead
    ↓
├─ 掉落物生成
├─ 任务完成
├─ 仇恨清除
└─ 统计更新
```

通用软件里：

```text
用户注销
    ↓
清理 Session
关闭 WebSocket
取消任务
释放资源
记录日志
更新在线状态
```

这就是扇出（Fan-out）问题。

#### 2. 合并（Merging）

多个事实共同决定一个新事件：

```text
Fact A
    \
     → Rule → Event C
    /
Fact B
```

它带来的问题是：**一个结果依赖哪些条件？**

| 游戏中 | 电商系统里 |
|---|---|
| `任务完成 + 进入新场景 → 触发Boss战` | `库存足够 + 支付成功 + 风控通过 → 订单成立` |

这对应复杂事件处理（CEP）中的模式匹配。

#### 3. 循环（Feedback Loop）

传播产生的新事件可能重新影响自身：

```text
A → B → C → A
```

它带来的问题是：**传播什么时候停止？**

| 游戏中 | 云原生容器里 |
|---|---|
| `伤害 → 反伤 → 伤害 → 反伤` | `服务故障 → 自动扩容 → 流量变化 → 负载变化 → 再次自动扩容` |

这是反馈系统的核心问题。因果层必须引入终止机制：时间窗口、次数上限、优先级抢占或资源限制。

#### 4. 抢占（Preemption）

新事件可能打断正在进行的传播链：

```text
技能释放 → 蓄力中 → 受到眩晕 → 取消技能
```

它带来的问题是：**过去的因果链是否仍然有效？**

这在长期运行系统中极其重要。例如：

| 系统 | 正在进行的传播链 | 抢占事件 | 结果 |
|---|---|---|---|
| 操作系统 | 进程运行 | 高优先级中断 | 抢占 CPU |
| 数据库 | 事务执行 | 检测死锁 | 回滚事务 |
| 游戏 | 死亡传播进行中 | 复活事件到达 | 中止死亡传播，优先处理复活 |

### 四个来源与核心性质的对应

这四种复杂性直接塑造了因果层必须具备的性质：

| 复杂性来源 | 核心问题 | 因果层需要的性质 |
|---|---|---|
| 分叉 | 传播范围失控 | 隔离边界、订阅机制，控制扇出影响范围 |
| 合并 | 依赖关系复杂 | 明确的依赖定义与裁决规则，给出唯一事实版本 |
| 循环 | 无法自然收敛 | 有界收敛性：时间窗口、次数上限、资源限制 |
| 抢占 | 历史路径被重新解释 | 可裁决性、优先级与生命周期管理 |

传统软件工程把这些困难拆成不同领域研究——并发、分布式一致性、事件系统、状态机、控制论、规则引擎——但它们本质上是同一个因果传播网络在不同约束下的表现。

## 核心性质

### 1. 可裁决性（Adjudicability）

对于冲突的事件：

```text
A 想修改 X
B 想修改 X
```

系统必须给出明确结果，不能长期处于「A 和 B 都可能成立」的状态。

### 2. 确定性（Determinism）

相同的历史事实、事件和规则，必须得到相同结果：

```text
History + Event + Rules
            ↓
        New History
```

### 3. 可追溯性（Traceability）

任何事实都有完整因果链：

```text
玩家攻击 → 命中判定 → 造成伤害 → 血量耗尽 → 死亡
```

而不是：

```text
Enemy.dead = true
```

无法追溯是谁导致的。

### 4. 有界收敛性（Bounded Convergence）

一次有限的因果计算必须结束。允许存在循环，但必须被时间窗口、优先级抢占或系统限制终止，不能无限传播。

---

## 最终因果一致性

因果层一旦做出裁决，系统就在有限时间内收敛到该裁决所确定的事实版本。这一要求的完整定义见[核心概念](/docs/dual-world-theory/core-concepts)。

长期运行的系统永远处于：

```text
外部事件 → 裁决 → 事实 → 传播 → 新事件 → ...
```

---

## 不同系统中的因果层

不同复杂软件都是这套机制在不同约束下的实现：

| 系统 | 事件 | 裁决规则 | 事实 |
|---|---|---|---|
| 游戏服务器 | 玩家操作 | 命中判定、伤害计算 | 权威世界状态 |
| 数据库 | 事务请求 | 隔离级别、约束检查 | 提交后的数据版本 |
| 操作系统 | 中断、系统调用 | 调度策略、权限检查 | 资源分配结果 |
| 区块链 | 交易广播 | 共识机制、签名验证 | 上链交易 |
| 交易所 | 订单提交 | 撮合规则、价格优先 | 成交记录 |

结构相同，约束不同。

---

## 一句话总结

> **因果层是世界事实的唯一生成通道。它通过「事件 → 裁决 → 事实 → 传播」的过程，使一个持续变化的系统在局部上不断形成稳定、可解释、可追溯的因果历史。**
