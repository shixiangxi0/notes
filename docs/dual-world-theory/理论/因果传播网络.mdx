# 因果传播网络

> **事件需要裁决，事实需要传播。**

因果层接收外部事件，将事件裁决为事实，再让事实传播产生新事件，最终形成可追溯的历史版本。

---

## 事件

* 因果层的基础单位，表示一个尚未被世界接受的变化请求。
* 可以来自外部输入，也可以来自内部传播。

游戏中：

```text
attack_requested
move_requested
spell_cast_requested
```

数据库中：

```text
UPDATE account SET balance = balance - 100
```

操作系统中：

```text
进程请求 CPU 时间片
设备发出中断
```

事件本身不是事实。例如两个玩家同时拾取一个物品：

```text
pick(A, item)  pick(B, item)
```

两个事件都进入系统，但不能同时成为事实。

---

## 事实

事实是事件经过裁决后，进入历史的关系变化。

```text
damage_applied
hp_depleted
entity_dead
trade_executed
```

事实具有三个特点：

- **权威性**：属于既定的不可变事实，可以往下覆盖，但是不能直接改写。
- **可追溯**：每一个事实都能回答「为什么发生」「由哪个事件触发」「经过哪些规则裁决」。
- **派生性**：事实派生状态，以及通过传播规则产生新的事件。

---

## 裁决与传播

因果层通过两类规则运行：

- **裁决规则**：决定事件能否成为事实。
- **传播规则**：决定事实会产生哪些新事件。

核心循环：

```text
事件 → 裁决 → 事实 → 传播 → 新事件 → ...
```

以游戏里的一次攻击为例：

```text
玩家按下攻击键
        ↓
attack_requested（事件）
        ↓
裁决规则：目标在范围内？技能冷却？敌人闪避成功？玩家攻击Buff加成？敌人防御力减伤？触发敌人反击？触发战吼？
        ↓
damage_applied（事实）
        ↓
传播：is_entity_dead (事件)
        ↓
裁决规则：hp 是否小于 0? 是否免疫致命伤害？是否可以复活？
        ↓
entity_dead（事实）
        ↓
传播：触发亡语、QTE、生成掉落物、更新任务、获取经验、触发升级、触发新剧情（新事件）
```

## 复杂性的根源：因果传播网络

如果没有时间，世界只是一个静态关系结构；一旦引入时间，世界就变成不断运行的因果传播网络：

```text
静态结构（无时间）                因果传播网络（有时间）

Object                            Event
  |                                 ↓
State                             Fact
  |                                 ↓
Relation                          Event
                                    ↓
                                  Fact
                                    ↓
                                   ...
```

左侧的复杂度只来自规模：对象多少、状态多少、关系多少。这是**空间复杂度**。

右侧的复杂度不再来自“有什么”，而来自：

> **变化如何传播。**

对象、状态、模块只是承载这些传播关系的空间结构。软件复杂，本质上是因为我们必须在持续变化的因果网络中管理四个问题：

- 一个变化会影响多少地方？
- 一个结果依赖哪些条件？
- 传播什么时候停止？
- 新的变化如何中断旧传播？

这也是为什么传统面向对象在这类系统中会显得吃力：OOP 擅长回答“谁拥有数据”、“谁属于谁”——这些都是静态空间关系；但复杂软件真正需要回答的是动态因果关系。

### 复杂度的来源

因果传播网络中，几乎所有的工程困难都可以归为基本拓扑问题。

#### 1. 分叉（Branching）

一个事实可能影响多个未来事件：

```text
Fact
  |
  +--> Event A
  |
  +--> Event B
  |
  +--> Event C
```

它带来的问题是：**一个变化会影响多少地方？**

游戏中：

```text
entity_dead
    ↓
├─ 掉落物生成
├─ 任务完成
├─ 仇恨清除
└─ 统计更新
```

前后端系统中：

```text
用户注销
    ↓
清理 Session
关闭 WebSocket
取消任务
释放资源
数据库清理数据
记录日志
更新在线状态
```

这就是扇出（Fan-out）问题。

#### 2. 合并（Merging）

多个事实共同决定一个新事件：

```text
Fact A
    \
     → Event C
    /
Fact B
```

它带来的问题是：**一个结果依赖哪些条件？**

| 游戏中 | 电商系统里 |
|---|---|
| `任务完成 + 进入场景 → 触发Boss战` | `库存足够 + 支付成功 + 风控通过 → 订单成立` |

这对应复杂事件处理（CEP）中的模式匹配。

#### 3. 循环（Feedback Loop）

传播产生的新事件可能重新影响自身：

```text
A → B → C → A
```

它带来的问题是：**传播什么时候停止？**

| 游戏中 | 云原生容器里 |
|---|---|
| `伤害 → 反伤 → 伤害 → 反伤` | `负载变化 → 自动扩容 → 流量变化 → 负载变化 → 再次自动扩容` |

这是反馈系统的核心问题。因果层需要引入终止机制：时间窗口、次数上限、优先级抢占和资源限制。

#### 4. 抢占（Preemption）

新事件可能打断正在进行的传播链：

它带来的问题是：**是否需要原子化事务，保障回退？**

这在长期运行系统中极其重要。例如：

| 系统 | 正在进行的传播链 | 抢占事件 | 结果 |
|---|---|---|---|
| 操作系统 | 进程运行 | 高优先级中断 | 抢占 CPU |
| 游戏 | 死亡事实传播进行中 | 复活事件到达 | 中止死亡传播，优先处理复活 |

#### 5. 延迟（Delay）

事实产生的影响可能发生在未来：

释放技能
   |
  0.5 秒
   ↓
命中判定

或者：

订单创建
   |
  30分钟
   ↓
订单超时

它带来的问题是：一个变化什么时候生效？

因此系统需要定时器、优先级抢占和生命周期管理。

#### 6. 竞争与顺序（Competition & Ordering）

多个事件可能竞争同一个未来事实：

```text
Event A
    \
     → Fact C
    /
Event B
```

所有竞争最终都需要形成一个明确的因果顺序。

因此系统需要消息队列，调度器，锁，原子操作。

**传统软件工程把这些困难拆成不同领域研究——并发、分布式、生命周期、中断机制、超时、控制反馈等等——但它们本质上是同一个因果传播网络在不同约束下的表现。**

## 核心性质

### 1. 可裁决性（Adjudicability）

对于冲突的事件：

```text
A 想修改 X
B 想修改 X
```

系统必须给出明确结果，不能悬而不决。

### 2. 确定性（Determinism）

相同的历史事实、事件和规则，必须得到相同结果。因果层维护的始终是权威事实，权威事实必须具有确定性。

### 3. 可追溯性（Traceability）

任何事实都有完整因果链：

```text
玩家攻击 → 命中判定 → 造成伤害 → 血量耗尽 → 死亡
```

决不能直接：

```text
Enemy.dead = true
```

无法追溯是谁导致的。

### 4. 收敛性（Convergence）

**任何有限范围内的因果传播必须在有限时间内结束。**

---

## 不同系统中的因果层

不同复杂软件都是这套机制在不同约束下的实现：

| 系统 | 事件 | 裁决规则 | 事实 |
|---|---|---|---|
| 游戏服务器 | 玩家操作 | 命中判定、伤害计算 | 权威世界状态 |
| 数据库 | 事务请求 | 隔离级别、约束检查 | 提交后的数据版本 |
| 操作系统 | 中断、系统调用 | 调度策略、权限检查 | 资源分配结果 |
| 区块链 | 交易广播 | 共识机制、签名验证 | 上链交易 |
| 交易所 | 订单提交 | 撮合规则、价格优先 | 成交记录 |

结构相同，约束不同。

---

## 总结

> **因果层是世界事实的唯一生成通道。它通过「事件 → 裁决 → 事实 → 传播」的过程，使一个持续变化的系统在局部上不断形成稳定、可解释、可追溯的因果历史。**
