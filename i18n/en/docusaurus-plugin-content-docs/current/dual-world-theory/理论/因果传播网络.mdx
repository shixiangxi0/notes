# Causal Propagation Network

> **Events need adjudication; facts need propagation.**

The Causal Layer is a metaphor for time, and events are its smallest unit.

```mermaid
flowchart LR
    E[Event] -->|Adjudicate| F[Fact]
    F -->|Propagate| NE[New Event]
    NE --> E
```

In this loop, events are input, facts are output, and propagation turns output into the next round of input. By running this cycle repeatedly, the Causal Layer turns the unordered changes of the outside world into a traceable and explainable history inside the system.

---

## Events and Facts

An **event** is a requested change that the world has not yet accepted. It may come from external input or from internal propagation within the Causal Layer.

A **fact** is a relational change that has been formally admitted into history after adjudication.

The difference between them is crucial:

- Events may conflict, be invalid, or be discarded.
- Once a fact is established, it is not rewritten in place; later changes enter history as new facts.

This means the Causal Layer does not maintain "current state"; it maintains a **continuously growing historical chain**. State is just the projection of that chain at the present moment.

```text
State model:      Causal model:
hp = 100         damage_taken
hp = 80          damage_taken
hp = 0           hp_depleted
                 entity_dead
```

> See also [Causal Layer](/docs/dual-world-theory/theory#causal-layer)

## Adjudication and Propagation

The Causal Layer runs on two kinds of rules:

- **Adjudication rules**: decide whether an event can become a fact.
- **Propagation rules**: decide which new events a fact produces.

Take an attack in a game:

```text
Player presses attack key
        ↓
attack_requested (event)
        ↓
Adjudication: target in range? skill off cooldown? enemy dodges? attack buff? defense mitigation? counterattack? battle cry?
        ↓
damage_applied (fact)
        ↓
Propagation: death_check_requested (event)
        ↓
Adjudication: hp < 0? immune to fatal damage? can resurrect?
        ↓
entity_dead (fact)
        ↓
Propagation: deathrattle, QTE, loot drop, quest update, experience gain, level-up, plot trigger (new events)
```

Adjudication and propagation are two sides of the same coin:

> **Adjudication collapses uncertainty into facts; propagation lets facts reopen uncertainty.**

## Causal Propagation Network

Without time, the world is just a static graph of relations: objects, states, and relations. Its complexity comes from scale—this is **spatial complexity**.

Once time enters, complexity no longer comes from "what exists," but from **"how change propagates."** Traditional object-oriented design asks spatial questions: who owns this data? What belongs to what? But complex software truly troubles us with temporal questions: **How does one change cause others? How can the world remain stable amid endless change?**

Thus, static structure becomes a **causal propagation network**.

The causal propagation network is the core data model of the Causal Layer. It can be seen as a causal graph that continuously generates change, handling six basic topologies:

### Six Basic Topologies

**1. Branching**

One fact affects multiple future events:

```text
    Fact
   /  |  \
  v   v   v
 A     B     C
```

**2. Merging**

Multiple facts jointly produce a new change:

```text
Fact A      Fact B
   \          /
    v        v
      Event C
```

**3. Feedback Loop**

Propagation influences itself:

```text
A → B → C
↑       |
└───────┘
```

**4. Preemption**

A new event interrupts an ongoing chain:

```text
A → B → C
    ↑
    X  (new event interrupts B→C)
```

**5. Delay**

The effect of a fact occurs in the future:

```text
Order created
    |
30 minutes
    ↓
Order timeout
```

**6. Competition and Ordering**

Multiple events shape one fact:

```text
Event A     Event B
   \           /
    v         v
      Fact C
```

### Topology-to-Engineering Mapping

| Causal Complexity | Engineering Manifestation | Typical Techniques |
|---|---|---|
| Branching | One change affects many modules | Event bus, pub-sub, Observer, message queues |
| Merging | Multiple conditions jointly determine behavior | State machines, rule engines, CEP, workflow |
| Feedback loop | Changes keep influencing each other | Feedback control, transaction boundaries, iteration limits, resource bounds |
| Delay | A change takes effect later | Timer, scheduler, job queue, cron, timeout |
| Preemption | New events interrupt old processes | Interrupts, cancellation tokens, rollback, state-machine switching |
| Competition and ordering | Multiple changes contest one outcome | Locks, CAS, transactions, consensus, schedulers |

**Objects, states, and modules are merely spatial containers of propagation paths. The seemingly unrelated fields of software engineering—concurrency, distributed systems, transactions, interrupts, timeouts, feedback control—are all manifestations of the same causal propagation network under different constraints.**

## Core Properties

For the Causal Layer to operate reliably, it must have four properties:

**Adjudicability**: Conflicts cannot remain unresolved forever; the system must produce a clear factual version.

**Determinism**: The same historical facts, events, and rules must yield the same result.

**Traceability**: Every fact can be embedded in a complete causal chain, not isolated as `Enemy.dead = true`.

**Convergence**: Any causal propagation within a finite scope must end in finite time.

Convergence is the most important of the four; the others exist mainly to guarantee it. It is the fundamental assurance that the world can run stably. See [Eventual Causal Consistency](/docs/dual-world-theory/theory#eventual-causal-consistency).

## Causal Layers Across Different Systems

| System | Event | Adjudication Rules | Fact |
|---|---|---|---|
| Game server | Player actions | Hit checks, damage rules | Authoritative factual version |
| Database | Transaction requests | Isolation levels, constraints | Committed data version |
| Operating system | Interrupts, syscalls | Scheduling, permission checks | Resource allocation outcome |
| Blockchain | Broadcast transactions | Consensus, signature verification | On-chain transaction |
| Exchange | Submitted orders | Matching rules, price priority | Trade record |

Same structure, different constraints.

---

## Summary

> **The Causal Layer is the only channel through which world facts are formed. Through the loop of Event → Adjudication → Fact → Propagation, it turns continuous change into stable, explainable, and traceable history.**

The hardest part of complex software is not objects and states, but **how change unfolds in time**. The causal propagation network offers a perspective: do not ask what the system contains; ask what happens in it, why it happens, and what consequences it brings.
