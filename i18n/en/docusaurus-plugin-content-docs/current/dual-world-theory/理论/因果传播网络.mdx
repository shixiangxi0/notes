# Causal Propagation Network

> **Events need adjudication; facts need propagation.**

The Causal Layer receives external events, adjudicates them into facts, then propagates those facts to produce new events, ultimately forming a traceable history.

---

## Events

- The basic unit of the Causal Layer: a requested change that has not yet been accepted as part of the world
- They may come from external input or from internal propagation

In games:

```text
attack_requested
move_requested
spell_cast_requested
```

In databases:

```text
UPDATE account SET balance = balance - 100
```

In operating systems:

```text
process requests CPU time slice
device issues interrupt
```

An event is not yet a fact. If two players try to pick up the same item:

```text
pick(A, item)  pick(B, item)
```

both events may enter the system, but both cannot become facts.

---

## Facts

A **fact** is a relational change that has been admitted into history after adjudication.

```text
damage_applied
hp_depleted
entity_dead
trade_executed
```

Facts have three important properties:

- **Authority**: once accepted, a fact is not rewritten in place; later change enters history as new facts
- **Traceability**: every fact can answer why it happened, which event triggered it, and which rules decided it
- **Derivability**: facts derive state and may trigger new events through propagation rules

---

## Adjudication and Propagation

The Causal Layer operates through two classes of rules:

- **Adjudication rules** decide whether an event can become a fact
- **Propagation rules** decide what new events a fact produces

Its core loop is:

```text
Event -> Adjudication -> Fact -> Propagation -> New Event -> ...
```

Take a simple attack in a game:

```text
player presses attack
        ↓
attack_requested (event)
        ↓
adjudication rules: target in range? skill off cooldown? dodge succeeds? buffs apply? defense reduces damage? counterattack triggered?
        ↓
damage_applied (fact)
        ↓
propagation: death_check_requested (event)
        ↓
adjudication rules: is hp below 0? fatal damage immunity? resurrection available?
        ↓
entity_dead (fact)
        ↓
propagation: deathrattle, QTE, loot drop, quest update, experience gain, level-up, new plot trigger
```

## The Real Source of Complexity

Without time, the world is only a static relational structure. Once time enters the picture, it becomes a continuously running causal propagation network:

```text
Static Structure (no time)        Causal Propagation Network (with time)

Object                            Event
  |                                 ↓
State                             Fact
  |                                 ↓
Relation                          Event
                                    ↓
                                  Fact
                                    ↓
                                   ...
```

The complexity on the left comes mostly from scale: how many objects, how many states, how many relations. That is **spatial complexity**.

The complexity on the right comes from something else:

> **how change propagates**

Objects, states, and modules are only the spatial containers of those propagation paths. Much of software complexity comes from managing a changing causal network:

- How many places does one change affect?
- What conditions does a result depend on?
- When does propagation stop?
- How does new change interrupt an older chain?

That is why traditional OOP often feels strained in these systems. OOP is good at answering static questions such as "who owns this data" or "what belongs to what." Complex software often needs answers to dynamic questions about causality.

### Structural Sources of Complexity

Many engineering difficulties in a causal propagation network can be reduced to a handful of recurring topologies.

#### 1. Branching

One fact may affect several future events:

```text
Fact
  |
  +--> Event A
  |
  +--> Event B
  |
  +--> Event C
```

The core question is: **how far does one change fan out?**

In games:

```text
entity_dead
    ↓
├─ loot spawn
├─ quest completion
├─ aggro clear
└─ statistics update
```

In service systems:

```text
user logout
    ↓
clear session
close WebSocket
cancel task
release resources
clean database state
record log
update online status
```

This is the **fan-out** problem.

#### 2. Merging

Several facts may jointly determine one new event:

```text
Fact A
    \
     → Event C
    /
Fact B
```

The core question is: **what does a result depend on?**

| In games | In e-commerce |
|---|---|
| `quest_complete + enter_scene -> trigger_boss_battle` | `inventory_available + payment_successful + risk_check_passed -> order_created` |

This is closely related to pattern matching in Complex Event Processing (CEP).

#### 3. Feedback Loops

Propagation may circle back and affect itself:

```text
A -> B -> C -> A
```

The core question is: **when does propagation stop?**

| In games | In cloud-native systems |
|---|---|
| `damage -> thorn_damage -> damage -> thorn_damage` | `load_change -> autoscale -> traffic_shift -> load_change` |

This is why the Causal Layer needs termination mechanisms such as time windows, count limits, preemption, and resource bounds.

#### 4. Preemption

New events may interrupt an ongoing propagation chain.

The core question is: **what must be interruptible, what must remain atomic, and what must support rollback?**

For example:

| System | Ongoing chain | Preempting event | Result |
|---|---|---|---|
| Operating system | Process execution | High-priority interrupt | CPU is preempted |
| Game | Death propagation in progress | Resurrection event arrives | Death propagation is aborted and resurrection takes priority |

#### 5. Delay

The effect of a fact may occur later:

```text
skill cast
   |
0.5 seconds
   ↓
hit check
```

Or:

```text
order created
   |
30 minutes
   ↓
order timeout
```

The core question is: **when does a change take effect?**

That is why systems need timers, scheduling, preemption, and lifecycle management.

#### 6. Competition and Ordering

Several events may compete for the same future fact:

```text
Event A
    \
     → Fact C
    /
Event B
```

All such competition must eventually collapse into a clear causal order.

In practice, these problems show up in message queues, schedulers, locks, atomic operations, and related mechanisms.

Once a system contains enough events, conflicts, timing races, and synchronization concerns, the need for a Causal Layer starts to emerge naturally. The question is not whether causality exists, but whether it is handled explicitly and centrally or implicitly and diffusely.

Traditional software engineering often separates these difficulties into different fields such as concurrency, distributed systems, lifecycle management, interrupts, timeouts, or feedback control. Underneath, they are all manifestations of the same causal propagation network under different constraints.

## Core Properties

### 1. Adjudicability

For conflicting events:

```text
A wants to modify X
B wants to modify X
```

the system must produce a result. It cannot leave the conflict permanently unresolved.

### 2. Determinism

The same facts, events, and rules should produce the same result. The Causal Layer maintains authoritative facts, so its outcomes must be deterministic.

### 3. Traceability

Every fact should sit inside a complete causal chain:

```text
player attacks -> hit check -> damage dealt -> hp depleted -> death
```

not merely:

```text
Enemy.dead = true
```

because that says nothing about who caused it.

### 4. Convergence

**Any causal propagation inside a finite scope should end within finite time.**

## Causal Layers Across Different Systems

Different kinds of complex software instantiate the same pattern under different constraints:

| System | Event | Adjudication Rules | Fact |
|---|---|---|---|
| Game server | Player actions | Hit checks, damage rules | Authoritative factual version admitted into history |
| Database | Transaction requests | Isolation levels, constraints | Committed data version |
| Operating system | Interrupts, syscalls | Scheduling, permission checks | Resource allocation outcome |
| Blockchain | Broadcast transactions | Consensus, signature verification | Confirmed on-chain transaction |
| Exchange | Submitted orders | Matching rules, price priority | Trade record |

Same structure, different constraints.

---

## Summary

> **The Causal Layer is the only channel through which world facts are formed. Through the loop of Event -> Adjudication -> Fact -> Propagation, a continuously changing system turns noise into stable, explainable, and traceable history.**
