# Preface

I keep returning to one question: do many of the difficulties in today's programming models originate in the von Neumann architecture itself? And if so, how could we make a genuinely fundamental improvement?

The von Neumann model is built around memory, state, instructions, and state transitions. Memory is space; data lives at addresses; programs read and write that space. At its core, this is a **spatial metaphor**.

More deeply, the von Neumann architecture encodes both instructions and data as contents of the same addressable memory. Instructions stand for time; data stands for space. The complexity of programming is then passed up through every layer: from the architecture to the operating system, and from the operating system to the programming language. The result is that **space becomes coupled with time**.

Take garbage collection. A GC algorithm traverses `Root → Object Graph`, keeps what is reachable, and deletes what is not. This is **inferring time from space**. But what if we thought differently? A piece of data disappears not when it becomes unreachable, but when its causal role is finished—as Event Sourcing and event-driven architectures already suggest. If we model systems as flows of events rather than as objects, the whole picture changes.

At a deeper level, we have been replacing **event + causal history** with **state + state transition**, and pushing the resulting complexity upward until it lands in the programming language, where it appears as object-oriented programming. OOP bundles state (space) with behavior (time). Every object becomes a small, self-contained loop with its own private timeline, and the state space of the whole system explodes.

I believe **time has been badly underestimated in modern programming models**. Time should not be merely a physical clock. It can itself be treated as a formalizable data structure. Events are not simply placed inside time; rather, the relationships between events *are* time. Time can be expressed as a causal model, and a causal model can be seen as a dynamic graph of propagation: A causes B, B causes C—the simplest serial causal chain is just one instance of this.

More broadly, software engineering is a projection of the real world. Time and space differ fundamentally in their mathematical and physical nature, and they must be handled separately. **Causality must be made explicit, and time and space must both be first-class citizens.**

Therefore, a running system must clearly separate two different things:

- **Causal runtime**: responsible for resolving conflicts in time—competition, ordering, propagation, lifecycle, consistency—and establishing a single history. It is the metaphor of time.
- **Perceptual runtime**: lets all spatial components unfold in parallel without races—scale, organization, nesting, rendering, resource management—and presents the already-determined history in space. It is the metaphor of space.

The two must not be confused, and neither can replace the other.

This is the core idea of Causality-Oriented modeling.
